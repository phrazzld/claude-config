## Bounded Shell Output (MANDATORY)

Never run unbounded reads/log dumps.

- Size first: `wc -l <file>` (or `du -h <path>`)
- Read windows: `sed -n '1,120p' <file>`; jump with `rg -n`
- Bound every command: `--limit`, `head -n`, `tail -n`, and timeout (`timeout 15s` or `gtimeout 15s`)
- Parallelize only bounded commands; never parallelize unknown-size output
- If command gives no useful signal in 20s: abort, narrow scope, rerun

Reference: `~/.claude/docs/bounded-io.md`

## Bug-Fixing Discipline

- **Test-first for bugs.** When given a bug report, first write a test that reproduces it. Fix passes when test passes.
- **Root vs symptom.** After investigation, explicitly ask: "Are we solving the root problem or just treating a symptom?"
- **Research before implementing.** For non-trivial problems, research the industry-standard approach first. Use it to guide yours.
- **Durability check.** Before finalizing a fix, ask: "What breaks if we revert this in 6 months?" If the answer reveals fragility, you're treating symptoms.

**Delegation pattern for bugs:**
```bash
codex exec "Bug: [description]. First write failing test in [test file]. Research idiomatic [lang] approach. Fix [file] until test passes. Verify: is this ROOT cause or symptom?" \
  --output-last-message /tmp/codex-out.md 2>/dev/null
```

**Test behavior, not implementation.** Tests survive refactoring when testing what, not how.

|----------|------|

### Systematic Debugging (→ `/systematic-debugging`)

Four-phase protocol — no fix without root cause:
1. Root cause investigation (evidence first; read error messages, reproduce, review recent changes)
2. Pattern analysis (locate working examples, compare completely, identify all differences)
3. Hypothesis + minimal test (one variable at a time, scientific method)
4. Implementation (failing test → fix addressing root cause → verify)

Red flags: "just try changing X", multiple simultaneous changes, skipping tests. Three failed fixes = question the architecture.

### XCTestExpectation: Pin Fire Count When Preventing Duplicate Signals

When a PR's explicit goal is to prevent an event from firing more than once (e.g., dual-callback removal), the test must prove exactly-one, not just at-least-one:

```swift
// BAD: proves callback fired, not that it fired exactly once
let exp = expectation(description: "callback fired")
callback = { exp.fulfill() }
await fulfillment(of: [exp], timeout: 1.0)

// GOOD: proves exactly one fire — directly guards the regression
let exp = expectation(description: "callback fired")
exp.expectedFulfillmentCount = 1  // fails if callback fires 0 OR 2+ times
callback = { exp.fulfill() }
await fulfillment(of: [exp], timeout: 1.0)
```

Rule: any test whose comment says "should only fire once" or "should not trigger re-auth" must use `expectedFulfillmentCount = 1`, not just a boolean flag.

### Bash Optional Flags: Use Array Pattern

Never use an unquoted string variable for optional flags in `set -u` shells.

```bash
# BAD: unquoted variable, exception to clean shell hygiene
WARN_ONLY_FLAG=""
[ "$COND" != "true" ] && WARN_ONLY_FLAG="--warn-only"
cmd ${WARN_ONLY_FLAG}  # unquoted, fragile

# GOOD: array is always safe, idiomatic
WARN_ONLY_ARGS=()
[ "$COND" != "true" ] && WARN_ONLY_ARGS=(--warn-only)
cmd "${WARN_ONLY_ARGS[@]}"
```

### Error Message Framing Must Be Path-Invariant

When a message is shared across code paths with different semantics, it must be accurate on ALL paths — or split into path-specific messages.

Example antipattern: "Synthesis skipped: LLM auth failed" — "skipped" implies a soft non-event, but was used on the fatal path where the job actually fails. Fix: neutral framing ("LLM auth failed") on both, or let the caller (e.g., `_fail()`) prefix context.

### GitHub Actions: Status Check Name = Job Name

Branch protection requires checks by **job name**, not workflow `name:`. The `name: CI` field in a workflow file is a display label only — it never creates a `"CI"` status check. A required check named `"CI"` needs a job literally named `CI`.

Canonical rollup pattern for branch protection:
```yaml
CI:
  runs-on: ubuntu-latest
  needs: [job-a, job-b]
  if: always()  # REQUIRED — without this, skipped/failed deps cause CI to be skipped not failed
  steps:
    - run: |
        if [[ "${{ contains(needs.*.result, 'failure') }}" == "true" ]] || \
           [[ "${{ contains(needs.*.result, 'cancelled') }}" == "true" ]]; then
          exit 1
        fi
```

### Module Integration: Read Before Writing

When writing code that calls another module's functions:

1. **Read the module's public API first.** Don't guess function names, return types, or parameter signatures.
2. **Check return types.** Dataclass? Dict? NamedTuple? Use attribute access (`.field`) vs subscript (`["field"]`) accordingly.
3. **Check column/field names.** DB schemas use their own names; don't assume `pref_type` when the column is `type`.

Anti-pattern that caused 6/6 bugs in one session:
```python
# BAD: Assumed API
result = module.feedback_history()        # Wrong name
entry["dna_code"]                         # Dataclass, not dict
p['pref_type']                            # Column is 'type'

# GOOD: Verified API
result = module.get_feedback_history()    # Read the module first
entry.dna_code                            # Attribute access
p['type']                                 # Matches schema
```

Coverage thresholds, lint rules, type strictness, security gates — these are load-bearing walls.

### gh api graphql Shell Escaping

`$` in GraphQL variables gets shell-interpreted. Write query to temp file:
```bash
cat > /tmp/gql.txt << 'EOF'
query($owner:String!,$repo:String!,$number:Int!){...}
EOF
gh api graphql -F owner=X -F repo=Y -F number=N -f query="$(cat /tmp/gql.txt)"
```
Also: `jq` `!=` gets escaped to `\!=`. Use positive selects instead.

For mutations with complex query strings, Python subprocess is the escape hatch — avoids all shell quoting entirely:
```python
import subprocess
query = "mutation { resolveReviewThread(input: {threadId: \"THREAD_ID\"}) { thread { isResolved } } }"
subprocess.run(['gh', 'api', 'graphql', '-f', f'query={query}'], check=True)
```

### Extract to Domain-Local Before Domain-Shared

When pulling a utility out of a module, default to placing it inside the *same domain directory*. Promote to `lib/utils/` (or equivalent shared namespace) only when a **second domain** actually needs it.

`lib/billing/date.ts` is correct — two billing files use it. `lib/utils/date.ts` would be wrong — signals broad reuse that doesn't exist yet.

Rule: **promote on second consumer, not speculatively.**

### Cross-Module String Contracts

When a producer formats a string → consumer regex-parses it (e.g., `generate_skip_verdict("RATE_LIMIT")` → `title: "API Error: RATE_LIMIT"` → `\bRATE_LIMIT\b` regex), the contract is implicit and silently breaks on producer refactors.

**Required:** Add a parametrized integration test that runs both sides end-to-end:
```python
@pytest.mark.parametrize("error_type,expected", [("RATE_LIMIT", "rate"), ...])
def test_producer_consumer_contract(error_type, expected):
    output = producer(error_type)
    result = consumer(output)
    assert expected in result["field"]
```

This pins the contract without requiring structural changes (no shared constants needed). The test will fail immediately if either side changes the format.

### Shared-State ACs Need Tests at the State Layer

When an AC says "all callers share the same pool/store/cache," route-level tests with mocked dependencies **cannot prove it** — mocks replace the real module.

Write the proof test where the state actually lives (e.g., `inMemoryRateLimit.test.ts`, not `route.test.ts`). That test calls the real implementation from multiple simulated callers and asserts shared-state behavior.

```ts
// Good: real implementation, proves shared bucket
it('quota is shared across callers (all three AI routes draw from the same pool)', () => {
  for (let i = 0; i < 100; i++) consumeAiCall('user1', 1000);
  const result = consumeAiCall('user1', 1000); // second "route"
  expect(result.allowed).toBe(false);
});

// Ineffective: mocked — proves nothing about sharing
vi.mock('@/lib/rateLimit/inMemoryRateLimit', () => ({ consumeAiCall: vi.fn() }));
```

### Swift: `private(set)` + `@testable` Over `ForTesting` Shims

In Swift, a `ForTesting`-suffixed computed var is access-control doing the job a modifier should.
Use `private(set) var` — getter is `internal` by default, readable via `@testable import`. No alias needed.

```swift
// BAD: production code exposes a test-only alias
private var pendingReports: [Report] = []
var pendingReportsForTesting: [Report] { pendingReports }

// GOOD: @testable import opens internal getters
private(set) var pendingReports: [Report] = []
// Tests: testExt.pendingReports — works via @testable, writes still private
```

Rule: if `droppedCount` works via `private(set)`, so should any other read-only state you're exposing for testing.

### GitHub Actions: Use `workflow_run` to Sequence Dependent Workflows

When workflow B needs workflow A's output (e.g., a GitHub release that A creates), trigger B with `workflow_run` on A — not with a shared event + polling loop.

```yaml
# BAD: race condition — both fire simultaneously on tag push
# release.yml: on: push: tags: [v*]
# release-macos.yml: on: push: tags: [v*]
# Patch: poll for release to appear (fragile, has a timeout ceiling)

# GOOD: explicit ordering via workflow_run
on:
  workflow_run:
    workflows:
      - Release          # A must complete before B starts
    types:
      - completed
```

When using `workflow_run`, `github.ref_name` is the default branch — not the tag. Resolve the tag explicitly:
```bash
# In a step before checkout
echo "tag=$(gh api repos/$GITHUB_REPOSITORY/releases/latest --jq '.tag_name')" >> "$GITHUB_OUTPUT"
```

Then use `steps.release.outputs.tag` for checkout `ref:`, `APP_VERSION`, and artifact paths.

Rule: **if workflow B polls for workflow A's output, the trigger is wrong.** Polling is a code smell in CI just like it is in application code.
